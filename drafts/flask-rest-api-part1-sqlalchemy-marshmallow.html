I just finished reading A Philosophy of Software Design by John Ousterhout. I learned how to decompose complex software systems into small components that can be implemented independently. Each chapter illustrates a design principle and goes into detail on how it improves overall system design. Hint: it's all about reducing complexity. Software Design is difficult and this book fills in a much needed knowledge gap.

Chapter 10 describes how to "Define Errors Out of Existence":

If a system processes a series of requests, it's useful to define an exception that aborts the current request, cleans up the system's state, and continues with the next request. The exception is caught in a single place near the top of the system's request-handling loop. This exception can be thrown at any point in the processing of a request to abort the request. ... Exception aggregation works best if an exception propagates several levels up the stack before it is handled; this allows more exceptions from more methods to be handled in the same place.

Been using Flask quite extensively over the past few months; love most things about the framework, but I found error handling a bit clunky. I didn't know what I wanted until I read the above passage, at which point I Googled "flask HTTP exception" and found the API Exception pattern.

Of course it's built-in! This is why we should read documentation. Can't use best practices if you don't know what they are. I'll take this a step further: if you import a library, you are responsible for its behaviour; you need to read (or, at least, skim) the source code of all your dependencies. It's what Pragmatic Programmers do.

In this post, we will build a REST API using Flask, SQLAlchemy, and Marshmallow. We'll also leverage the API Exception pattern to consolidate error handling logic and reduce boilerplate code.
