Solving Problems Using Code

Mon 20 November 2017

lessons learned at work

Posted by Aly Sivji in backburner   

So this was a really fun week at work. After spending the first week getting our development environment set up with docker compose and containers, we finally started to write some code.

On Monday I sat down with the senior engineer I'm working with to figure out how to plan out the next sprint.

We're building a system from scratch so I wanted to make sure all my bases are covered and what not.

We broke down my task and i had the idea of separating out middleware

I'm working on writing out logging middleware and an authorization service. While rwriting the middleware, I had the smart idea of making it into a package so we can reuse it across all the services.

There are arguments in if this is the best way to proceed.

Kent Beck First make it work THen make it right THen make it fast and small

This is what the senior engineer on the team says

Doing it this way introduced huge headaches that I didn't quite know how to deal with. Leaving me scrambling for the best solution through seeing what other people did.

THis is a problem I think a lot of people in CS experience, we assume others know more than us and that we can follow their steps and produce the same results. But this is not how thigns work. There are a lot of moving pieces and not everybody is doing it the right way.

Best solution is to take a step back. Breathe. That's super important. Figure out how best to proceed. Read the documentation.

I always talk about my love of StackOverflow, but also about how much we've come to rely on it as a crutch. Can't figure it out? Ask a question and see if somebody else did it and we can just copy and paste it into what we're doing

This is now how to proceed. We should start by ooking at the documentation. I talked about this back when I started this blog and really started programming in general. list to post

Poeple might not know the best way of doing it. As I learned. I should've just researched how to do it and I could've figured out a solution which is a lot more acceptable for our workflow instead of trying to adapt something.
