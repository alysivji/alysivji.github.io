Testing 101: An Introduction

Posted by Aly Sivji in Drafts   

  • Intro Blurb

    • I've been interested in testing since I read about it earlier this year. There was something about adding structure to programming that appealed to my need for having structure.
    • It took me a few days to weeks to research and try out different techniques until I was able to get a grasp on what is going on
    • There are a lot of good sources out there. Lots of advice, sometimes conflicting. This post is my way of trying to sort out my thoughts about testing as I apply it to a project in which I start writing unit tests around a microservices framework I am building for a practice side project
    • first of a series of posts on testing and tests

There is something about testing that just speaks to me. Maybe it's because good tests make it so you know if software is working, but I'm a huge fan of it.

  • high level overview of why tests are important and why we should always test our code

    • mostly we test so we can sleep at night (have picture of me / emoji sleeping in talk)
    • A Case for Testing
      • Bugs will happen, you can bank on it, only the severity of their impact will vary.
      • Bug incidents can increase dramatically as new features are added
      • Bug fix times can increase dramatically as new features are added
      • Software testing is NOT designed to make sure code works when itâ€™s written but rather to ensure future changes do not break existing code.
      • A well run, tightly managed testing regime leads to more stable, reusable code.

  • brief description of the different types of testing

    • unit testing
      • focus of first post
      • book: working effectively with unit test
    • integration testing
      • how to do this?
    • functional testing / system testing
      • pyresttest
      • selenium
    • manual testing / acceptance / UAT / whatever else is in tao of testing
    • testing pyramid
    • from The Tao of Testing
      • unit test - Test a single public method or function
      • black box test - test provides data and expects certain results without knowing how those results are derived
      • integration test - many parts of the system are tested at once, tests a complete end-to-end lifecycle of an application or a portion versus testing selected components independently
      • manual test - test completed by human
      • load/scale test - impose stress on application by simulating heavy load conditions / highly concurrent load conditions
      • infrastructure test - testing infrastructure underlying software
      • QA test - like manual tests
      • UAT (user acceptance testing) - human test assess implementation of software to see if it meets specifications

  • My thoughts

    • I don't really have a horse in the race so I'm gonna test using common sense and figure out my own style as I advance in my career as a Software Engineer. Right now I think testing is awesome. I believe in TDD, as in test while you are developing. It can be first or right after. It's mostly to save you time later on.

    • For now I have to test a microservice REST API and need to be pragmatic (?) about how I do things. In the next few posts, I will explore the subject of testing in more detail in the context of the above project

    • also i use a dynamically typed language (Python) which makes it a lot easier to monkeypatch. I feel like i dont have to deal with a lot of the formality of other languages

    • basically my philsophy of testing is to be pragmatic. test what is required. err on the side of testing to much. and test driven development means that testing is driving the development of your architecture, i.e. you are thinking about tests as you write code. it isn't really about testing first or testing last. it's about testing being a part of your thought process.

  • Random Thoughts

    1. I don't like absolutes, but if you don't test your code you are not a developer. You are a hacker. Once you start testing, you are on your way to becoming a craftsman. learning how to test is starting a journey from "hacker to craftsman"
    2. DHH talking about how TDD is a cult and that programming best practices are just pseudoscience. I don't agree because being methodical about how you do things helps makes things easier in the long run. This isn't to say that you can't hack stuff together to solve problems. If all you care about is solving problems, you can do that. But there is also a reason that monliths needs to be made into microservices. It becomes a mess. You worry a lot about if things are going to break upon refactoring. I think test driven doesn't really mean test first. It means test first or test closely after. He argues about dependency injection. I mean, if your language doesn't support monkeypatching... you need to inject dependencies (but we'll come back to this in later post). Thinks unit tests are broken b/c we need system tests instead. I mean, can't we have both. Unit tests to ensure that we don't regress and system tests to ensure we meet what we are supposed to do. double loop TDD is the way forward. Functional testing! But he ends it well. Not sure if I agree with everything, but I do like his pragmatic approach to testing
