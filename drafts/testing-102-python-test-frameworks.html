Testing 102: Python Test Frameworks

Posted by Aly Sivji in Deep Dives   

Status: Draft

In order to understand the principles of testing, we have to understand the underlying theory about testing. This was covered in Testing 101: An Introduction to Testing. In this post, we will delve into Test Frameworks.

(change) It's still highly theoretical, but we have a lot of ground to cover. This foundation will help you think about tests in a deeper way. Know that we know the theory behind testing, we can start looking into tools that will help us test more efficiently. While we can definitely write tests, it helps to have something that does the work for us. Start talking about test frameworks and how they are useful.

Definitions

There is a lot of testing specific terminology that we should cover to make sure we are all on the same page.

Test Fixture

Recall that one of the properties of tests is that they should be deterministic. This means we want to run our tests against a known environment to ensure we can results are repeatable.

Test fixtures set up our test environment to a known state before running a given test case (setup stage). After the test is run, we return our test environment back to its original state (teardown stage). Running setup before and teardown after our test ensure deterministic results.

Test Case

As discussed in Testing 101, we can add structure to our tests using one of the Arrange-Act-Assert or Given-When-Then patterns.

This means that each of our tests will have the following distinct phases:

  • set up pre-conditions
  • run test
  • check post-conditions are met

A test case is an individual unit (different word?) of test that follows the above pattern.

Test Suite

A test suite is a collection of test cases. Depending on what we are trying to accomplish, we can run our entire test suite or a focused subset of our tests. If we are fixing a bug or adding a feature, running a small subset of tests can provide a faster feedback loop during the development process. Once our change is complete and we see that our local subset of tests is passing, we should run the entire test suite before committing the code to version control.

More and more organizations are adopting Continuous Integration and Continuous Deployment pipelines which require the entire test suite to pass before code can be merged into master or deployed into your staging or production environment.

Test Runner

The test runner orchestrates test execution and results reporting. Test failures are indicated with stack trace and variable introspection to provide insights on next steps.

Invariant

Invariants are conditions that are always True.

In Python, we use the assert statement to evaluate invariants:

  • If the statement evaluates to True, the program continues
  • If the statement evaluates to False, an AssertionError is raised

Test Framework

Recall that one of the properties of tests is that they should be automated. We want to be able to test software as quickly and as thoroughly as possible. Once the code base starts to grow, manual testing becomes a time consuming process that is not able to discover every defect. The only way we know our program works is by testing it. Test automation provides a short feedback cycle and gives us confidence that our changes work and do not break existing functionality.

A test framework is an execution environment for automated tests. It falls under the category of Developer Tools that increase productivity.

Test Framework Features

Defined Way of Writing Tests

Like with every kind of framework, you have to function within the bounds defined by the framework. Every framework has it's own way of setting up and tearing down fixtures and verify post-conditions.

By following the parameters defined by the framework, it is able to hook into and drive the System Under Test (SUT).

Test Runner

When you run your tests: the framework will go out find all the tests you have written; this process is called test discovery. As the test framework runs through the test suite, it tracks which tests pass and which tests fail.

Results Reporting

After all the tests are run, the framework will display a report detailing the pass/fail status for each test that was run along with a stack trace to provide contextual information around the failure.

Test Frameworks in Python

There are two main test frameworks in Python: unittest and pytest.

unittest

✅ Part of Standard Library
✅ Familiar xUnit-style test pattern
✅ Extensive documentation
❌ Lots of Boilerplate code
❌ Feels like Java

pytest

✅ Doesn’t feel like Java
✅ Tests are easy to read and write
✅ Extensive documentation
✅ Plugin architecture
✅ Runs unittest test suites

pytest

For all the reasons above, pytest is my framework of choice. In this section, we will explore pytest.

Features

Writing Tests

Unlike unittest which forces you to write a lot of boilerplate, pytest tests look like standard Python functions:

def test_calculating_aggregate_statistics_happy_path():
    """pytest example test

    - Function name should be descriptive
    - Function should start with "test_"
    - Complicated tests should have a docstring detailed steps
    """
    # Arrange
    record = Record("details", True)
    db.session.add(record)
    db.session.commit()

    # Act
    result = get_aggregate_stats()

    # Assert
    assert result is True

While you can also use classes to define your tests and have pytest pick them up, it feels cleaner to write tests without all that pageantry. There was a great talk at PyCon 2012 by Jack Diederich titled Stop Writing Classes which I highly recommend everybody watch.

Assertions

pytest does a little bit of work behind the scenes to allow checking of post-conditions using the built-in assert statement. This makes tests easy to read and easy to write.

pytest also provides detailed introspection on assertions failures:
